# 跨领域拆解 SOP 模板

> 用途：用于从零到一拆解任意领域的优质样本，并沉淀出可复用的分析框架、质量标准和产出模板
> 适用对象：博主分析、竞品拆解、行业研究、课程拆解、账号诊断、内容风格分析、产品案例分析
> 核心目标：把“我觉得这个东西做得好”翻译成“我知道它为什么好、怎么复用、最低标准是什么”

---

## 一、先判断：这次拆解要解决什么问题

在开始前，先写清楚 4 个问题：

| 问题 | 你要填写的内容 |
|------|----------------|
| 拆解对象是什么？ | 例如：小红书 AI 博主 / SaaS 官网 / 知识付费课程 / 招聘 JD |
| 你想得到什么？ | 例如：内容公式 / 选题库 / 结构模板 / 质量标准 / 机会点 |
| 最终给谁用？ | 例如：自己创作 / 团队交付 / AI Skill / 新人培训 |
| 这次最重要的判断是什么？ | 例如：能否复用 / 哪些是核心优势 / 最低质量线在哪 |

**产出要求**：用 3 句话以内写清楚本次拆解任务，不要一上来就收集材料。

---

## 二、输入材料清单

至少准备以下材料中的 3 类：

1. 优质样本 3-10 份
2. 样本来源链接、截图、原文、转录稿或导出数据
3. 你自己的目标对象信息
4. 行业背景资料
5. 你已有的草稿、判断或问题清单

推荐做一个材料登记表：

| 样本名 | 类型 | 来源 | 为什么选它 | 备注 |
|-------|------|------|------------|------|
| 样本A | 报告/账号/页面 | 链接或路径 | 明显比普通样本高一个档次 | |
| 样本B | 报告/账号/页面 | 链接或路径 | 有结构、有判断、有启发 | |

**最低标准**：
- 不要只看 1 个样本
- 不要只收“同一种风格”的样本
- 样本必须是你真心认可的，不是为了凑数

---

## 三、拆解总流程

```text
明确任务
  ↓
收集样本
  ↓
逐份拆解
  ↓
横向找共性
  ↓
提炼结构
  ↓
提炼质量标准
  ↓
区分脚本/AI/人工分工
  ↓
写出最小可用模板
  ↓
试跑 1-3 次
  ↓
修订成稳定版本
```

---

## 四、逐步执行 SOP

### Step 1：定义拆解任务

**目的**：避免拆着拆着变成“信息搬运”。

**你要做的事**：
- 写清这次要拆的对象、范围和边界
- 写清最终要交付什么文档
- 写清哪些问题必须回答

**交付物**：
- 《任务定义》1段话
- 《必须回答的问题》3-5条

**模板**：

```md
### 任务定义
本次要拆解的是【对象】。
目标不是泛泛介绍，而是产出【报告类型】。
这份报告要回答：1）___ 2）___ 3）___ 4）___。
```

---

### Step 2：选样本

**目的**：找到真正值得学习的标杆，不要被“看起来还行”的样本带偏。

**筛选标准**：
- 有明确结构
- 有数据或证据
- 有判断，不只是描述
- 有启发，不只是总结
- 有可执行建议或可迁移方法

**你要做的事**：
- 先粗筛 10-20 个
- 再精筛 3-10 个进入正式拆解
- 给每个样本写一句“为什么入选”

**交付物**：
- 《标杆样本表》

---

### Step 3：逐份样本拆解

**目的**：先看单个样本“怎么写得好”，不要急着找共性。

对每个样本都回答下面 8 个问题：

| 维度 | 要回答的问题 |
|------|--------------|
| 目标 | 这份样本想解决什么问题？ |
| 读者 | 它是写给谁看的？ |
| 结构 | 它分成了哪些章节？ |
| 重点 | 哪些章节最值钱？为什么？ |
| 证据 | 它用什么支撑观点？数据、案例、评论、截图？ |
| 判断 | 它在哪些地方有明确结论？ |
| 建议 | 它怎么把分析翻译成行动？ |
| 风格 | 它为什么读起来不空？ |

**交付物**：
- 每个样本 1 张《单样本拆解卡》

**单样本拆解卡模板**：

```md
## 样本名称：

### 1. 它解决的问题

### 2. 它的目标读者

### 3. 它的章节结构

### 4. 最有价值的部分

### 5. 它如何提供证据

### 6. 它有哪些明确判断

### 7. 它给了哪些可执行建议

### 8. 我想保留什么
```

---

### Step 4：横向找共性

**目的**：把“个体写得好”变成“这类产出物应该怎么写”。

重点找 4 类共性：

1. 结构共性
2. 判断共性
3. 证据共性
4. 表达共性

推荐做一个共性提炼表：

| 观察项 | 样本A | 样本B | 样本C | 共同规律 |
|--------|-------|-------|-------|----------|
| 开头怎么写 | | | | |
| 核心章节是什么 | | | | |
| 怎么下判断 | | | | |
| 怎么给建议 | | | | |
| 怎么做收尾 | | | | |

**关键动作**：
- 不要只统计“出现了几次”
- 要判断“哪些共性是真正决定质量的”

**交付物**：
- 《共性归纳结论》5-10条

---

### Step 5：提炼“分析视角”

**目的**：确定以后每次拆解都从哪些角度切入。

常见视角包括：
- 对象全貌视角：它是谁、做了什么、表现如何
- 方法公式视角：哪些方法能复用
- 选题机会视角：下一步可以做什么
- 结构数据视角：数据和规律是什么
- 用户需求视角：用户到底在意什么
- 差异化竞争视角：它和同行比强在哪

**你要做的事**：
- 选 3-5 个对你最有用的视角
- 给每个视角写一句“它回答什么问题”

**交付物**：
- 《分析视角定义表》

---

### Step 6：搭建标准结构

**目的**：把视角变成稳定章节，而不是每次临场发挥。

推荐按下面的方式写：

| 章节 | 回答什么问题 | 最低要求 |
|------|--------------|----------|
| 概览 | 这是什么对象，基本情况如何 | 有基础信息 + 一句判断 |
| 重点拆解 | 哪些案例最值得看 | 至少 3 个案例，含证据和原因 |
| 模式总结 | 共性方法是什么 | 至少 3 条共性 |
| 用户启示 | 对我们有什么用 | 至少 3 条可执行建议 |
| 核心结论 | 最后记住什么 | 1 句总结 + 1 个公式 |

**注意**：
- 先写“最低结构”，不要一开始就追求大而全
- 章节数 5-8 个最稳

**交付物**：
- 《产出结构 1.0》

---

### Step 7：把“质量感”翻译成检查项

**目的**：解决“我知道它好，但我说不清为什么”的问题。

建议从下面 8 个维度提炼质量标准：

1. 有明确问题，不跑题
2. 有结构，不散
3. 有观点，不空
4. 有证据，不虚
5. 有对比，不平
6. 有因果，不浅
7. 有建议，不止观察
8. 有轻重，不平均用力

**质量检查表模板**：

| 检查项 | 合格标准 | 常见不合格表现 |
|--------|----------|----------------|
| 是否有明确结论 | 每章至少 1 条明确判断 | 只有描述，没有结论 |
| 是否有数据锚点 | 关键观点后有数字/案例/原话 | “表现不错”“用户喜欢” |
| 是否有因果解释 | 能解释“为什么会这样” | 只说结果，不说原因 |
| 是否有行动建议 | 至少 3 条具体建议 | “建议继续优化内容” |
| 是否有层次 | 重点章节展开，次重点概括 | 每章都差不多长 |

**交付物**：
- 《最低质量标准表》

---

### Step 8：区分脚本、AI、人工的分工

**目的**：让这个流程以后能复用、能放大、能交给别人做。

通用分工参考：

| 工作类型 | 更适合谁做 | 说明 |
|----------|------------|------|
| 基础信息整理 | 脚本/人工 | 结构化提取 |
| 数据统计 | 脚本 | 确定性计算 |
| 章节骨架 | AI/人工 | 可模板化 |
| 模式归类 | 脚本初筛 + AI复核 | 先规则后判断 |
| 因果解释 | AI/人工 | 需要语境理解 |
| 建议生成 | AI/人工 | 需要结合目标 |
| 终稿审核 | 人工 | 负责最后判断 |

**交付物**：
- 《分工说明》

---

### Step 9：写出最小可用模板

**目的**：先有能跑的版本，再慢慢迭代。

最小模板建议只保留这 5 部分：

1. 目标
2. 必答问题
3. 固定结构
4. 质量标准
5. 分工说明

**交付物**：
- 《最低下限质量标准 1.0》

---

### Step 10：试跑与修订

**目的**：检验模板是否真的能用。

至少试跑 1-3 次，重点看：
- 哪些章节最容易空
- 哪些标准太抽象，没法判断
- 哪些要求太重，执行成本过高
- 哪些地方缺少例子

每轮试跑后更新：
- 保留什么
- 删除什么
- 新增什么

**交付物**：
- 《版本修订记录》

## 五、最小交付模板

如果你只想先做一个最低可用版本，可以直接复制下面这份：

```md
# 【领域/对象】拆解最低质量标准

## 1. 目标
本文档用于拆解【对象】，目的是产出【报告类型】，帮助【使用者】解决【核心问题】。

## 2. 必须回答的问题
1. 这个对象的基本情况是什么？
2. 它最值得关注的部分是什么？
3. 它为什么有效/成功/突出？
4. 哪些方法可以复用？
5. 对我们下一步有什么启示？

## 3. 固定结构
### 3.1 概览
- 基础信息
- 一句话判断

### 3.2 重点案例拆解
- 至少 3 个案例
- 每个案例包含：现象 / 证据 / 原因 / 启示

### 3.3 模式总结
- 至少 3 条共性
- 指出最关键的一条

### 3.4 行动建议
- 至少 3 条
- 必须具体可执行

### 3.5 核心结论
- 1 句核心观点
- 1 个公式化总结

## 4. 合格标准
- 不能只有描述，必须有判断
- 不能只有判断，必须有证据
- 不能只有现象，必须有原因
- 不能只有分析，必须有行动建议
- 不能平均用力，重点部分必须展开

## 5. 分工
- 数据整理：
- 初步归类：
- 深度分析：
- 最终审核：
```

## 六、通用提示词模板

如果你要把这套 SOP 交给 AI，可以直接用这段：

```text
你现在要做的不是泛泛总结，而是按“跨领域拆解SOP”输出一份高质量拆解报告。

任务目标：
1. 先识别拆解对象的核心问题
2. 再对提供的样本做逐份拆解
3. 横向提炼共同结构、共同写法和共同判断
4. 输出一个适用于该领域的最低质量标准框架

输出要求：
1. 不要只描述内容，要解释为什么这样设计
2. 每个重要观点后尽量附证据、案例或原话
3. 必须有“核心发现”或“结论”段落
4. 必须有“对用户/团队的启示”
5. 最后输出：
   - 固定结构
   - 最低质量标准
   - 可复用模板
   - 建议脚本化与建议AI处理的部分

如果信息不足，请明确指出缺什么，不要硬编。
```

## 七、复盘问题清单

每次做完一轮拆解，都用下面这些问题复盘：

1. 这次拆解有没有真正回答核心问题？
2. 哪些章节最值钱，哪些章节只是凑数？
3. 哪些判断是有证据支撑的，哪些只是感觉？
4. 哪些建议别人拿去就能做，哪些还是太空？
5. 这套结构下次还能不能直接复用？
6. 有没有哪些部分其实应该脚本化？
7. 这次最值得沉淀成固定规则的是什么？

---

## 八、你以后可以这样用这份模板

### 用法 A：快速开新领域
- 先找 3-5 份该领域好样本
- 直接按本 SOP 的 Step 1-7 跑一遍
- 先产出最低标准 1.0，不求完整

### 用法 B：给 AI 当生产规范
- 把“固定结构 + 质量标准 + 提示词模板”一起喂给 AI
- 让 AI 先出初稿
- 你只负责补判断、纠偏和定最终标准

### 用法 C：给团队做培训材料
- 把“单样本拆解卡”和“质量检查表”独立出来
- 让每个人先拆同一个样本
- 再统一对齐“什么叫高质量”

---

## 九、一句话原则

> 拆解不是把别人做了什么抄下来，而是把别人为什么做得好、哪些能复用、最低标准是什么，沉淀成以后还能重复使用的框架。

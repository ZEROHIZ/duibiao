# Bug 档案记录

## 1. Windows 临时视频文件删除失败 (WinError 32)
- **发现时间**：2026-06-22
- **问题描述**：在 Windows 系统上运行转换脚本时，转录完成后删除临时视频文件报错：`[WinError 32] 另一个程序正在使用此文件，进程无法访问。`
- **问题根源**：`transcribe_video` 函数在使用 `requests.post` 上传文件时，直接执行了 `open(video_path, 'rb')`，但没有在使用后显式关闭文件句柄，导致句柄被 Python 进程持有，Windows 系统拒绝删除该文件。
- **解决方案**：使用 `with open(video_path, 'rb') as f:` 上传视频，确保请求发送完成后自动释放文件句柄。

## 2. Whisper 转录请求读取超时 (Read timeout) 与动态超时需求
- **发现时间**：2026-06-22
- **问题描述**：转录长视频时，客户端极易报错：`HTTPConnectionPool: Read timed out.`。硬编码固定超时值无法兼顾短视频效率与超长视频的执行安全。
- **问题根源**：转录计算所需时长和视频长度成正比。短视频转录极快，而长视频（如几十分钟）需要很长时间，在 CPU 模式下处理耗时更长，原有的 300 秒超时会频繁中断。
- **解决方案**：实现免第三方依赖的纯 Python MP4 视频时长快速解析器，读取 MP4 mvhd 盒子得到精确时长（秒）。然后，根据视频时长动态计算超时时间：`超时时间 = 视频秒数 * 1.5 + 120 秒` (最少 120 秒)。这实现了短视频迅速失败、长视频安全放行的动态超时机制。

## 3. deep_analyze.py 在详情数据直接嵌套 note 时提取正文失败
- **发现时间**：2026-06-23
- **问题描述**：传入 details JSON 运行 deep_analyze.py 时，无法正确提取正文内容，导致正文长度、CTA 等数据分析结果均显示为 0。
- **问题根源**：`note = item.get("data", {}).get("note", item)` 逻辑在 details 中包含 `note` 键而无 `data` 键时，错误地回退到了 `item`，未能解包 `item["note"]`，导致之后访问 `desc` 等属性失败。
- **解决方案**：判断如果 `note` 直接在 `item` 中且为字典类型，优先使用 `item["note"]`。

## 4. “作品总览时间线”子页签切换时博主列表依然可见
- **发现时间**：2026-07-07
- **问题描述**：在“对标灵感”页面点击“最新作品流总览”时，虽然作品时间线显示了出来，但是博主监控管理列表仍然展示在下方，两者同时可见，导致排版混乱。
- **问题根源**：`blogger-list-view` 容器只添加了 `active-subview` 类，未添加 `subview` 类。在切换页签的 JS 逻辑中，切换子视图是用 `document.querySelectorAll(".subview").forEach(...)` 来移除/添加 `active-subview` 类的。由于 `blogger-list-view` 缺少 `subview` 类，未能被 `querySelectorAll` 选中，导致其高亮状态无法被移除。
- **解决方案**：在 `index.html` 的 `blogger-list-view` div 容器上增加 `subview` 类名，即 `class="subview active-subview"`。

## 5. 重复爬取导致 Whisper API 重复转录慢
- **发现时间**：2026-07-07
- **问题描述**：再次运行爬虫脚本时，已经爬取过的视频仍然被重新完整处理并尝试发起视频转录，导致运行开销很大且极慢。
- **问题根源**：
  1. 爬虫脚本没有加载已有的本地数据文件，即使视频 ID 已经存在，仍然会慢速执行 Hover 和评论点击操作并写入已有的 URL 链接到 `douyin_data.json`。
  2. 格式转换脚本只检测输入的 `desc` 字段是否为链接，但每次爬虫都会写入 URL 链接，导致转换脚本重新下载视频并重复发起 Whisper 请求。
- **解决方案**：
  1. 在 `douyin_crawler.py` 中引入 `existing_vids` 过滤。对于已存在的视频，跳过分享 Hover 和评论 API 拦截检测，直接用 `page.url` 快速校验 `ArrowDown` 翻页。
  2. 在 `convert_douyin_notes.py` 中，读取已存在的处理文件，如 `desc` 已经是解析出的文本，则复用该文本以彻底跳过语音转录请求。

## 6. 新建博主点不进详情页 & 系统配置页三个按钮点击无反馈
- **发现时间**：2026-07-09
- **问题描述**：
  1. 新增/未同步博主在博主监控列表中无法点击进入详情页（显示为“数据未同步”普通文本且点击无反应）。
  2. 系统参数配置页面中的“保存系统设置”、“一键更新全部博主”、“立即扫描转录”三个按钮点击后，控制台报错 `ReferenceError: showToast is not defined`，没有任何弹窗或跳转的视觉反馈。
- **问题根源**：
  1. 表格/卡片渲染时，硬编码了 `hasDistilled`（`total_notes > 0`）才能绑定跳转详情事件及渲染“蒸馏拆解”按钮。导致新录入的博主由于作品数为 0，根本无法打开详情页进行单博主的手动“同步更新数据”操作。且由于作品数为 0，详情页内计算占比时存在 `0/0*100` 的除零错误导致显示 `NaN%`。
  2. 弹窗消息通知函数 `showToast` 被局部声明在 `setupEventListeners()` 内部，导致全局作用域中的其它 API 提交及编辑处理函数（如 `handleSyncAllClick`、`loadBloggersList` 等）在调用 `showToast` 时直接抛出 ReferenceError 异常。
- **解决方案**：
  1. 移除 `total_notes > 0` 对进入详情页的限制，允许所有博主都可以点击进入详情页。并在详情页渲染时，对 `total_notes` 做大于 0 的判断，避免视频/图文比例计算出现除零错误显示 `NaN%`。
  2. 将 `showToast` 从 `setupEventListeners()` 内提到外层的全局作用域中，使其在 `app.js` 内的任意函数都可直接调用，保障点击事件的回调函数执行顺畅并有弹窗反馈。

## 7. “立即扫描转录”按钮在无转录任务时自动跳转且无有效反馈
- **发现时间**：2026-07-09
- **问题描述**：点击“立即扫描转录”后，即使数据库中目前没有任何需要转录的视频（即所有视频已转录完成或无视频数据），页面仍然会强制跳转至任务日志页，但该页面显示“暂无任务记录”，给用户造成“没有反应/功能失效”的错觉。
- **问题根源**：
  1. 后端接口 `/api/transcribe/trigger` 在被调用时，只设置了触发事件，并没有检索数据库中实际待转录视频的个数，从而直接返回了 success，前端无法得知实际是否有待处理的任务。
  2. 前端在收到 success 回应后，无条件直接将视图强制重定向到日志监控页。由于此时后端没有新增运行中的任务，日志页面便只能显示空空如也的“暂无任务记录”。
- **解决方案**：
  1. 修改后端接口 `/api/transcribe/trigger`，在触发前，先在 SQLite 中使用 SQL 语句 `SELECT COUNT(*) FROM blogger_notes WHERE type='video' AND (desc LIKE 'http://%' OR desc LIKE 'https://%' OR desc LIKE '[转录失败_第%')` 实时统计等待转录的视频数，并将数量 `count` 写入 JSON 响应中返回给前端。
  2. 修改前端点击回调函数 `handleTranscribeNowClick`，接收后端返回的 `count`：
     * 若 `count > 0`，则弹出 Toast 提示检测到有 X 个待转录任务并照常自动跳转到日志监控页；
     * 若 `count == 0`，则弹出 Toast 友好提示“后台扫描已完成，当前没有待转录视频（已全部转录完成或无视频数据）”，且保持在当前系统设置页面，不触发自动跳转。

## 8. Web 端更新博主数据流水线缺失 `analyze.py` 步骤导致数据无法增量录入
- **发现时间**：2026-07-09
- **问题描述**：通过 Web 看板点击“同步更新数据”或“一键更新全部博主”后，任务显示执行成功，但数据库中完全没有更新最新抓取的视频及笔记内容，依然是老旧的缓存数据。
- **问题根源**：
  1. 后端进行 SQLite 导入时运行的 `importer.py` 脚本是基于 `data/` 目录下的 `<博主名>_analysis.json` 作为主笔记列表来解析导入的。
  2. 但在 Web 看板的异步同步流水线 `scripts/pachopngjiaoben/pipeline.py` 中，只执行了 Phase 1（网页爬虫抓取 `douyin_crawler.py`）和 Phase 2（格式转换 `convert_douyin_notes.py`），漏掉了特征统计分析生成（`analyze.py`），直接执行了 Phase 3（`importer.py`）。
  3. 由于没有运行 `analyze.py` 重新生成 `<博主名>_analysis.json`，`importer.py` 每次运行都只能读取到本地上一次或冷启动自带的陈旧 `_analysis.json`，因此完全忽视了在 `_notes_details.json` 中已更新的最新抓取视频，导致数据库中总笔记数和明细内容原封不动。
- **解决方案**：
  * 修改 `scripts/pachopngjiaoben/pipeline.py`，在 Phase 2（格式转换）和 Phase 3（数据库导入）之间插入 Phase 2.5 数据深度特征分析生成步骤，命令行调用 `analyze.py` 读取刚生成的 `_notes_details.json` 并计算汇总至 `_analysis.json`。

## 9. 格式转换脚本直接覆盖已转换的 notes_details.json 导致历史/种子数据丢失与指标回退
- **发现时间**：2026-07-09
- **问题描述**：对已有的博主（如内置了 4 条种子数据的“小A”）在只爬取 2 条最新视频时，完成同步后其总获赞数、视频条数等分析指标直接回退缩水为仅 2 条视频的值，而原有的其它 2 条视频在明细列表里存在，但无法参与占比与均值统计。
- **问题根源**：
  1. 网页爬虫抓取的 raw 原始 JSON 文件是通过增量合并逻辑写入 `douyin_data.json` 的。
  2. 但负责格式规整的 `convert_douyin_notes.py` 脚本在读取 raw 数据并进行处理后，虽然读取了旧的 output 文件加载已转录文本映射关系，但最后直接将当前新转换的列表 `converted_list`（仅含本次抓取条数的记录）通过 `json.dump` 覆盖写入了 `_notes_details.json`，遗弃了历史在 `_notes_details.json` 中的旧笔记（包括冷启动附带的 4 条种子视频明细）。
  3. 导致后续 `analyze.py` 和 `importer.py` 在基于该 `_notes_details.json` 分析和覆盖博主主表时，汇总出的 `stats` 数据仅代表当前的这 2 条记录，从而出现总数缩水与历史笔记在主表指标中被“抹去”的逻辑错误。
- **解决方案**：
  * 修改 `convert_douyin_notes.py`，在主函数中获取完转换后的新列表 `converted_list` 后，将其与已读出的历史记录列表 `existing_list` 根据主键唯一标志符 `_feed_id` 实施字典增量合并去重，生成完整累计的 `final_list`。
  * 将 `final_list` 作为最终输出存盘回写到 `_notes_details.json`，并将其传入后续的数据质量校验模块中，确保后续的特征分析与导入流程能够累积式计算所有历史与新增数据。

## 11. 服务器重启/调用导入器时重新覆盖已转录文本导致视频反复转录
- **发现时间**：2026-07-09
- **问题描述**：视频明明已经转录完成（可以在后台日志或前端面板中看到转录出来的文本），但一旦重启后端 FastAPI 服务器（或者拉起一次导入流程），视频转录后台 Worker 又会重新扫描到待转录任务并开始重复转录。
- **问题根源**：
  1. 后端 `app.py` 在 `startup` 启动钩子中，执行了 `run_full_import()` 重新读取本地 `data/` 目录下的 `<博主名>_analysis.json`（其中对于视频，`desc` 依然保存为原始的 `http` 直链链接），并将其导进 SQLite。
  2. 导入逻辑 `importer.py` 中的 `INSERT OR REPLACE` 冲突更新逻辑没有做数据合并或防御检查，而是简单粗暴地用 `_analysis.json` 和 `_notes_details.json` 中的字段（包含直链 URL）去覆盖已有的 `blogger_notes` 数据库记录。
  3. 导致之前异步转录好的正文文字在数据库记录中被重置成了 `http://...` 直链。于是启动后，转录 Worker 扫库时又认为这是“待转录”状态，开始重新调用 Whisper API。
  * 增加合并安全判断：若原先数据库记录已存有转录出来的文字（即不是 `http://` 链接也不是 `[转录失败_第` 的重试标记），则在写入时，保留使用原先数据库里的成功转录数据，禁止将其覆盖重置。

## 12. 语音转录成功后未重算博主语义统计特征导致详情页展示陈旧
- **发现时间**：2026-07-09
- **问题描述**：视频虽然成功调用 Whisper 转录完成了，但在 Web 详情页的“写作模式与公式”、“认知层观点句”、“高频词分析”等标签页中，依然看不到这几个新转录视频包含的金句、观点、价值词，依然是未转录状态下的空洞/原始指标。
- **问题根源**：
  1. 系统在爬取同步时会采用 `--skip-transcribe` 快速入库。此时导入器计算了标题模式、形式占比等数据，但认知特征（观点句、高频词、分类汇总）是直接从不包含转录文本的 `<博主名>_analysis.json` 中读取并写入 `blogger_distilled` 表的。
  2. 当后台语音转录 Worker 异步转录成功后，虽然更新了 `blogger_notes` 中的 `desc` 正文，但**从未触发任何数据库重算任务**。
  3. 导致 `blogger_distilled` 中的核心语义特征（观点句、价值词等）永远停留在未转录状态下（全空或极少状态），无法展示转录正文带来的语义资产。
- **解决方案**：
  * 新建特征重算工具 [recalculate.py](file:///d:/daima/codex/蒸馏/blogger-distiller-main/scripts/utils/recalculate.py)，完全基于数据库 `blogger_notes` 里的最新正文与交互数据，重新运行分类算法、提取观点句、分析写作结构、计算高频词，并同步回写到 `bloggers` 和 `blogger_distilled` 表中，同时自动回写更新本地的 `<博主名>_analysis.json`。
  * 分别在导入器文件保存末尾以及 `app.py` 的后台转录 Worker 转录成功后（即 `success` 分支中），调用并运行该重算工具，实现整个系统的完全“数据库化”与“动态重算”，使得转录出来的文本瞬间与大盘、详情页和后续任务底稿无缝融合。
